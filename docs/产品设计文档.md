# MiniMeeting 产品设计文档

## 1. 产品概述

### 1.1 产品定位

MiniMeeting 是一个轻量级实时音视频通信系统，支持多人语音/视频通话，具备低延迟、弱网适应能力，并提供基础会议管理能力。

### 1.2 目标用户

- 需要低延迟音视频通话的小型团队
- 面试场景下展示 WebRTC 通信系统架构能力的开发者
- 对实时通信原理有学习需求的技术人员

### 1.3 核心能力

| 能力 | 说明 |
|------|------|
| 实时语音通话 | 基于 Opus 编码，支持低延迟语音传输（目标 <300ms） |
| 实时视频通话 | 基于 VP8/H264 编码，支持动态分辨率适配 |
| 多人房间 | 支持 2~4 人同时通话，Mesh 拓扑 |
| NAT 穿透 | STUN 优先直连，TURN 中继兜底 |
| 弱网适应 | 丢包补偿、动态码率、带宽自适应 |
| 延迟监控 | 实时显示 RTT、丢包率、码率等网络指标 |

### 2.2 数据流

```
[音频源] → 采集 → 编码(Opus) → SRTP → 网络
                                          ↓
[扬声器] ← 渲染 ← 解码(Opus) ← SRTP ← 网络

[摄像头] → 采集 → 编码(VP8) → SRTP → 网络
                                          ↓
[显示窗口] ← 渲染 ← 解码(VP8) ← SRTP ← 网络
```

## 3. 核心模块设计

### 3.1 设备采集层

**职责：** 从系统获取音视频原始数据。

| 子模块 | 说明 | 技术方案 |
|--------|------|----------|
| 音频采集 | 麦克风输入 | WebRTC 内置 ADM（Audio Device Module） |
| 视频采集 | 摄像头输入 | WebRTC 内置 VDM（Video Device Module） |

**音频参数：**
- 采样率：48kHz
- 声道：单声道
- 采样格式：16-bit PCM

**视频参数：**
- 分辨率：640x480（默认）/ 320x240（弱网降级）
- 帧率：30fps（默认）/ 15fps（弱网降级）

### 3.2 编解码层

**职责：** 对原始音视频数据进行编码和解码。

| 媒体类型 | 编解码器 | 特点 |
|----------|----------|------|
| 音频 | Opus | 低延迟、高压缩率、自适应码率（6~510 kbps） |
| 视频 | VP8 | 开源免版税、WebRTC 默认编码器 |
| 视频（可选） | H264 | 硬件加速支持好，需考虑许可证 |

**音频编码参数：**
- 比特率：32kbps（语音）~ 64kbps（音乐）
- 帧长：20ms
- 复杂度：5（平衡质量与延迟）

### 3.3 传输层

**职责：** 加密传输、拥塞控制、丢包恢复。

WebRTC 内置传输协议栈，关键机制如下：

| 机制 | 说明 |
|------|------|
| SRTP | 安全实时传输协议，对 RTP 数据加密 |
| RTP | 实时传输协议，承载音视频数据 |
| RTCP | 实时传输控制协议，反馈网络质量 |
| NACK | 负面确认，请求重传丢失的包 |
| FEC | 前向纠错，发送冗余数据防止丢包 |
| GCC | Google 拥塞控制，基于延迟梯度估计带宽 |

### 3.4 会话管理模块

**职责：** 管理 WebRTC 连接生命周期，是项目的核心业务逻辑。


### 3.5 UI 层

**职责：** 提供用户交互界面。

**页面结构：**

| 页面 | 功能 |
|------|------|
| 登录页 | 输入用户名，连接信令服务器 |
| 房间列表页 | 创建房间、查看可用房间、加入房间 |
| 通话页 | 音视频通话界面、控制栏（静音/关摄像头/挂断）、网络状态面板 |

**控制栏功能：**
- 麦克风开关
- 摄像头开关
- 扬声器开关
- 挂断（离开房间）
- 网络质量指示

## 4. 技术亮点设计

### 4.1 音频优先策略

**核心思路：** 先保证语音通话质量稳定，再叠加视频能力。

**音频优化方案：**

| 优化项 | 实现方式 | 效果 |
|--------|----------|------|
| 丢包补偿（PLC） | Opus 编码器内置 | 10% 丢包率下语音可懂 |
| Jitter Buffer | WebRTC NetEQ | 平滑网络抖动，减少卡顿 |
| 降噪 | RNNoise（可选） | 降低环境噪声 |
| 回声消除 | WebRTC AEC 内置 | 消除扬声器回声 |
| 自动静音检测 | WebRTC VAD 内置 | 无语音时不发送，节省带宽 |

### 4.2 弱网优化

**动态适配策略：**

```
网络质量良好
    │
    │  丢包率上升 / RTT 增大
    ▼
降低视频分辨率（640x480 → 320x240）
    │
    │  仍然不稳定
    ▼
降低视频帧率（30fps → 15fps）
    │
    │  仍然不稳定
    ▼
关闭视频，仅保留音频
    │
    │  仍然不稳定
    ▼
降低音频码率（64kbps → 32kbps）
```

**实现方式：**
- 监听 RTCP 报告中的丢包率、RTT 等指标
- 通过 `RTCStatsObserver` 获取实时网络质量数据
- 使用 `RtpParameters` 动态调整编码参数

### 4.3 延迟监控

**监控指标面板：**

| 指标 | 来源 | 说明 |
|------|------|------|
| RTT | RTCP Receiver Report | 往返延迟 |
| 丢包率 | RTCP Receiver Report | 丢包百分比 |
| 抖动（Jitter） | RTCP Receiver Report | 延迟波动 |
| 发送码率 | RTCOutboundRTPStreamStats | 上行带宽使用 |
| 接收码率 | RTCInboundRTPStreamStats | 下行带宽使用 |
| 帧率 | RTCOutboundRTPStreamStats | 视频发送帧率 |
| 分辨率 | RTCOutboundRTPStreamStats | 视频当前分辨率 |
| ICE 连接类型 | RTCIceCandidatePairStats | host/srflx/relay |

**UI 展示：** 通话界面底部浮动面板，实时刷新（每秒更新）。

### 4.4 多人房间（Mesh 拓扑）

**拓扑结构：** 每个参会者与房间内所有其他参会者建立独立的 PeerConnection。

```
    A ←───→ B
    ↕         ↕
    C ←───→ D
    ↕         ↕
    E ←───→ ...
```

| 项目 | 说明 |
|------|------|
| N 人房间 | 每人维护 N-1 条 PeerConnection |
| 上行带宽 | 发送 N-1 份音视频流 |
| 建议上限 | 4 人（带宽与性能平衡） |
| 扩展方向 | 后续可升级为 SFU 架构突破人数限制 |

## 5. 关键技术指标

| 指标 | 目标值 |
|------|--------|
| 音频延迟 | < 150ms（局域网）/ < 300ms（公网） |
| 视频延迟 | < 200ms（局域网）/ < 500ms（公网） |
| 音频抗丢包 | 10% 丢包率下语音可懂 |
| NAT 穿透率 | > 85%（含 TURN 兜底） |
| 最大参会人数 | 4 人（Mesh 拓扑） |
| 音频编解码 | Opus，32~64 kbps |
| 视频编解码 | VP8，300~2000 kbps |
