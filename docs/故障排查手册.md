### 1.2 日志级别说明

| 级别 | 说明 | 示例 |
|------|------|------|
| **DEBUG** | 调试信息，详细执行流程 | `DEBUG: ICE candidate collected: ...` |
| **INFO** | 正常运行状态信息 | `INFO: Connected to signaling server` |
| **WARNING** | 警告信息，可能存在问题 | `WARNING: High packet loss detected: 15%` |
| **ERROR** | 错误信息，功能受影响 | `ERROR: Failed to create peer connection` |
| **CRITICAL** | 严重错误，程序可能崩溃 | `CRITICAL: Out of memory` |

## 2. 连接问题排查

### 2.1 信令服务器连接失败

#### 现象
- 无法创建/加入会议
- 提示 "连接服务器失败"
- 界面一直显示 "连接中..."

#### 诊断步骤

```
步骤1: 检查服务器状态
─────────────────────────
$ curl -I https://signal.minimeeting.com/health
HTTP/1.1 200 OK  ✓ 服务器正常

步骤2: 检查 WebSocket 连接
─────────────────────────
浏览器控制台:
var ws = new WebSocket('wss://signal.minimeeting.com');
ws.onopen = () => console.log('连接成功');
ws.onerror = (e) => console.log('连接失败', e);

步骤3: 检查防火墙
─────────────────────────
$ telnet signal.minimeeting.com 443
Connected to signal.minimeeting.com.  ✓ 端口可达

步骤4: 检查证书
─────────────────────────
$ openssl s_client -connect signal.minimeeting.com:443
Verify return code: 0 (ok)  ✓ 证书有效
```

#### 常见原因与解决方案

| 原因 | 解决方案 |
|------|----------|
| **服务器未启动** | 检查服务器进程状态，重启服务 |
| **端口被占用** | 检查端口占用：`netstat -tlnp \| grep 443` |
| **SSL 证书过期** | 更新 SSL 证书 |
| **防火墙阻断** | 开放 443 端口，检查安全组规则 |
| **DNS 解析失败** | 检查 DNS 配置，尝试使用 IP 直连 |
| **代理设置问题** | 检查系统代理设置，或配置代理 |

#### 日志分析

```log
# 正常连接日志
[2024-01-15 10:30:15.123] [INFO] [SignalingClient::connect:45] Connecting to wss://signal.minimeeting.com
[2024-01-15 10:30:15.456] [INFO] [SignalingClient::onConnected:78] Connected to signaling server

# 连接失败日志
[2024-01-15 10:30:15.123] [INFO] [SignalingClient::connect:45] Connecting to wss://signal.minimeeting.com
[2024-01-15 10:30:20.124] [ERROR] [SignalingClient::onError:156] Connection timeout
[2024-01-15 10:30:20.125] [DEBUG] [SignalingClient::onError:158] Error details: QAbstractSocket::NetworkError
```

### 2.2 WebRTC 连接建立失败

#### 现象
- 成功加入会议但无视频/音频
- 状态显示 "连接中..."
- 远程用户看不到/听不到自己

#### 诊断步骤

```
步骤1: 检查 ICE 连接状态
─────────────────────────
chrome://webrtc-internals → 查看 PeerConnection 状态

状态流程：
new → checking → connected → completed
            ↓
         failed (问题)

步骤2: 检查 ICE 候选
─────────────────────────
查看是否收集到候选：
- host candidate (本地 IP)
- srflx candidate (公网 IP via STUN)
- relay candidate (TURN 中继)

步骤3: 检查 NAT 类型
─────────────────────────
使用工具检测 NAT 类型：
https://webrtc.github.io/samples/src/content/peerconnection/trickle-ice/

步骤4: 检查防火墙
─────────────────────────
需要开放的端口：
- STUN: UDP 3478
- TURN: UDP/TCP 3478, 5349 (TLS)
- 媒体端口范围: UDP 10000-20000
```

#### ICE 状态分析

```javascript
// 监控 ICE 状态变化
pc.oniceconnectionstatechange = () => {
    console.log('ICE state:', pc.iceConnectionState);
    
    switch (pc.iceConnectionState) {
        case 'checking':
            // 正在检查连接
            break;
        case 'connected':
            // 连接成功
            break;
        case 'completed':
            // 连接完成（包括 TURN）
            break;
        case 'failed':
            // 连接失败 - 需要排查
            console.error('ICE connection failed');
            // 检查 ICE 候选
            pc.getStats().then(stats => {
                stats.forEach(report => {
                    if (report.type === 'local-candidate') {
                        console.log('Local candidate:', report);
                    }
                });
            });
            break;
        case 'disconnected':
            // 临时断开，可能恢复
            break;
        case 'closed':
            // 连接已关闭
            break;
    }
};
```

#### 常见原因与解决方案

| 原因 | 症状 | 解决方案 |
|------|------|----------|
| **无 ICE 候选** | 只有 host 候选或无候选 | 检查 STUN/TURN 服务器配置 |
| **对称 NAT** | 无法 P2P 穿透 | 配置 TURN 中继服务器 |
| **防火墙阻断 UDP** | 无法建立媒体通道 | 开放 UDP 端口或使用 TCP/TLS |
| **IP 地址泄露保护** | 浏览器隐藏本地 IP | 配置 mDNS 或使用公网 IP |
| **ICE 超时** | 长时间 checking 状态 | 增加超时时间，检查网络 |

### 2.3 ICE 候选收集问题

#### 现象
- ICE 状态一直为 `checking`
- 只有本地候选，无远程候选
- 只收集到 host 候选

#### 诊断代码

```javascript
// 详细日志收集 ICE 候选
pc.onicecandidate = (event) => {
    if (event.candidate) {
        console.log('ICE candidate:', event.candidate);
        console.log('  Type:', parseCandidateType(event.candidate.candidate));
        console.log('  Protocol:', parseProtocol(event.candidate.candidate));
        console.log('  Address:', parseAddress(event.candidate.candidate));
    } else {
        console.log('ICE gathering complete');
    }
};

function parseCandidateType(candidate) {
    if (candidate.includes('typ host')) return 'host';
    if (candidate.includes('typ srflx')) return 'srflx';
    if (candidate.includes('typ prflx')) return 'prflx';
    if (candidate.includes('typ relay')) return 'relay';
    return 'unknown';
}
```

#### 候选类型说明

```
┌────────────────────────────────────────────────────────────┐
│                    ICE 候选类型                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  host      本地网络接口地址                                 │
│  ─────    优先级最高，延迟最低                               │
│           示例: 192.168.1.100                              │
│                                                            │
│  srflx     服务器反射地址（通过 STUN 获取）                   │
│  ─────    公网 IP，优先级高                                  │
│           示例: 203.0.113.5 (via STUN)                     │
│                                                            │
│  prflx     对等反射地址（NAT 映射）                          │
│  ─────    NAT 穿透成功后获得                                 │
│           示例: 203.0.113.10 (via NAT)                     │
│                                                            │
│  relay     中继地址（TURN 服务器）                           │
│  ─────    优先级最低，延迟最高                               │
│           示例: 198.51.100.1 (via TURN)                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 解决方案

```cpp
// 完善 ICE 配置
webrtc::PeerConnectionInterface::RTCConfiguration config;

// STUN 服务器
webrtc::PeerConnectionInterface::IceServer stun;
stun.urls.push_back("stun:stun.l.google.com:19302");
stun.urls.push_back("stun:stun1.l.google.com:19302");
config.servers.push_back(stun);

// TURN 服务器（必须配置用于对称 NAT）
webrtc::PeerConnectionInterface::IceServer turn;
turn.urls.push_back("turn:turn.minimeeting.com:3478");
turn.urls.push_back("turns:turn.minimeeting.com:5349");  // TLS
turn.username = "username";
turn.password = "password";
config.servers.push_back(turn);

// ICE 传输策略
config.type = webrtc::PeerConnectionInterface::kAll;

// 候选网络策略
config.network_preference = webrtc::PeerConnectionInterface::kNetworkPreferenceAny;

// 增加超时时间
config.ice_candidate_pool_size = 10;
```

## 3. 音频问题排查

### 3.1 无声音问题

#### 现象
- 对方听不到自己的声音
- 自己听不到对方的声音
- 双向无声音

#### 诊断流程

```
┌─────────────────────────────────────────────────────────────┐
│                    无声音诊断流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ 本地无声音（对方听不到自己）                            │  │
│  └──────────────────────────────────────────────────────┘  │
│       │                                                     │
│       ├─► 检查麦克风是否被静音                               │
│       │                                                      │
│       ├─► 检查麦克风权限                                    │
│       │      浏览器: 隐私设置 → 麦克风权限                   │
│       │      桌面: 系统设置 → 隐私 → 麦克风                  │
│       │                                                      │
│       ├─► 检查设备选择                                      │
│       │      是否选择了正确的麦克风？                        │
│       │                                                      │
│       ├─► 检查音轨状态                                      │
│       │      track.enabled = true?                          │
│       │      track.readyState = 'live'?                     │
│       │                                                      │
│       └─► 检查音频是否发送                                  │
│              WebRTC stats → audio outbound → bytesSent       │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ 远程无声音（自己听不到对方）                            │  │
│  └──────────────────────────────────────────────────────┘  │
│       │                                                     │
│       ├─► 检查扬声器音量                                    │
│       │                                                      │
│       ├─► 检查扬声器设备选择                                │
│       │                                                      │
│       ├─► 检查音轨是否接收                                  │
│       │      WebRTC stats → audio inbound → bytesReceived   │
│       │                                                      │
│       ├─► 检查是否被解码                                    │
│       │      audio inbound → decodedSamplesCount            │
│       │                                                      │
│       └─► 检查音频元素状态                                  │
│              <audio> 元素是否 muted?                         │
│              autoplay 是否启用?                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 诊断代码

```javascript
// 检查音频轨道状态
async function diagnoseAudio(pc) {
    const stats = await pc.getStats();
    
    stats.forEach(report => {
        // 音频发送统计
        if (report.type === 'outbound-rtp' && report.kind === 'audio') {
            console.log('音频发送:', {
                bytesSent: report.bytesSent,
                packetsSent: report.packetsSent,
                codecId: report.codecId
            });
            
            if (report.bytesSent === 0) {
                console.error('❌ 音频未发送，检查麦克风');
            } else {
                console.log('✓ 音频正常发送');
            }
        }
        
        // 音频接收统计
        if (report.type === 'inbound-rtp' && report.kind === 'audio') {
            console.log('音频接收:', {
                bytesReceived: report.bytesReceived,
                packetsReceived: report.packetsReceived,
                packetsLost: report.packetsLost,
                jitter: report.jitter
            });
            
            if (report.bytesReceived === 0) {
                console.error('❌ 未收到音频数据');
            } else if (report.packetsLost > 0) {
                console.warn('⚠ 音频丢包:', report.packetsLost);
            } else {
                console.log('✓ 音频正常接收');
            }
        }
    });
}
```

#### 常见原因与解决方案

| 原因 | 解决方案 |
|------|----------|
| **麦克风权限被拒绝** | 在系统设置中授予权限 |
| **错误的设备选择** | 在设置中选择正确的麦克风 |
| **麦克风被静音** | 取消静音，检查 `track.enabled` |
| **音频轨道未添加** | 检查 `addTrack()` 调用 |
| **编解码器不匹配** | 检查 SDP 中的编解码器协商 |
| **音频元素未自动播放** | 确保 `<audio autoplay>` 或调用 `play()` |

### 3.2 音频卡顿/断续

#### 现象
- 声音断断续续
- 声音像机器人
- 偶尔无声

#### 诊断

```javascript
// 检查音频质量指标
async function checkAudioQuality(pc) {
    const stats = await pc.getStats();
    
    stats.forEach(report => {
        if (report.type === 'inbound-rtp' && report.kind === 'audio') {
            // 计算丢包率
            const totalPackets = report.packetsReceived + report.packetsLost;
            const lossRate = report.packetsLost / totalPackets;
            
            console.log('音频质量分析:', {
                '丢包率': (lossRate * 100).toFixed(2) + '%',
                '抖动': report.jitter + ' 秒',
                '隐藏样本': report.concealedSamples,  // PLC 隐藏的样本数
                '插入样本': report.insertedSamplesForDeceleration
            });
            
            if (lossRate > 0.05) {
                console.error('❌ 高丢包率，可能导致卡顿');
            }
            if (report.jitter > 0.05) {
                console.warn('⚠ 高抖动，可能导致断续');
            }
            if (report.concealedSamples > 0) {
                console.log('PLC 已启用，隐藏丢包影响');
            }
        }
    });
}
```

#### 解决方案

```cpp
// 优化音频配置
AudioConfig config;

// 启用音频处理功能
config.echoCancellation = true;      // 回声消除
config.noiseSuppression = true;      // 噪声抑制
config.autoGainControl = true;       // 自动增益
config.highPassFilter = true;        // 高通滤波

// 启用 FEC（前向纠错）
config.fec.enabled = true;
config.fec.redundancy = 2;  // 冗余包数量

// 优化 Jitter Buffer
config.jitterBuffer.maxTargetDelay = 500;  // ms
config.jitterBuffer.minTargetDelay = 20;   // ms
config.jitterBuffer.acceleratedPlayback = true;
```

### 3.3 回声问题

#### 现象
- 对方听到自己的回声
- 持续的嗡嗡声

#### 原因分析

```
┌────────────────────────────────────────────────────────────┐
│                    回声产生原因                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 声学回声                                               │
│     ┌─────────┐      ┌─────────┐                          │
│     │ 扬声器  │─────►│ 麦克风  │                          │
│     │ (对方)  │ 声波  │ (本地) │                          │
│     └─────────┘      └─────────┘                          │
│     解决: 启用 AEC (回声消除)                               │
│                                                            │
│  2. 线路回声                                               │
│     ──────────────────────────────                         │
│     由于音频线路设计问题导致信号反馈                         │
│     解决: 使用外置声卡或高质量麦克风                        │
│                                                            │
│  3. 软件回声                                               │
│     系统音频设置中"监听此设备"被启用                        │
│     解决: 取消"监听此设备"选项                              │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 解决方案

```cpp
// 配置回声消除
AudioProcessingConfig config;

// AEC (Acoustic Echo Cancellation)
config.echoCanceller.enabled = true;
config.echoCanceller.mobileMode = false;  // 桌面模式
config.echoCanceller.exportLinearAECOutput = false;

// 残留回声抑制
config.residualEchoDetector.enabled = true;

// 噪声抑制（辅助）
config.noiseSuppression.enabled = true;
config.noiseSuppression.level = NoiseSuppressionLevel::kHigh;
```

### 5.1 网络状态监控

```javascript
// 持续监控网络状态
class NetworkMonitor {
    constructor(pc) {
        this.pc = pc;
        this.prevStats = {};
    }
    
    async collect() {
        const stats = await this.pc.getStats();
        const networkInfo = {};
        
        stats.forEach(report => {
            if (report.type === 'candidate-pair' && report.state === 'succeeded') {
                // 计算实时带宽
                const now = Date.now();
                const elapsed = (now - this.prevStats.timestamp) / 1000;
                
                if (this.prevStats.bytesReceived) {
                    const bytesReceivedDelta = report.bytesReceived - this.prevStats.bytesReceived;
                    networkInfo.downloadBitrate = (bytesReceivedDelta * 8 / elapsed / 1000).toFixed(0);
                }
                
                if (this.prevStats.bytesSent) {
                    const bytesSentDelta = report.bytesSent - this.prevStats.bytesSent;
                    networkInfo.uploadBitrate = (bytesSentDelta * 8 / elapsed / 1000).toFixed(0);
                }
                
                networkInfo.rtt = (report.currentRoundTripTime * 1000).toFixed(0);
                networkInfo.availableBitrate = (report.availableOutgoingBitrate / 1000).toFixed(0);
                
                this.prevStats = {
                    timestamp: now,
                    bytesReceived: report.bytesReceived,
                    bytesSent: report.bytesSent
                };
            }
            
            if (report.type === 'inbound-rtp') {
                const totalPackets = report.packetsReceived + report.packetsLost;
                if (totalPackets > 0) {
                    networkInfo[report.kind + 'Loss'] = 
                        (report.packetsLost / totalPackets * 100).toFixed(2);
                }
            }
        });
        
        return networkInfo;
    }
}

// 使用
const monitor = new NetworkMonitor(pc);
setInterval(async () => {
    const info = await monitor.collect();
    console.log('网络状态:', info);
}, 1000);
```

```cpp
double getCpuUsage() {
#ifdef _WIN32
    static FILETIME prevSysIdle, prevSysKernel, prevSysUser;
    FILETIME sysIdle, sysKernel, sysUser;
    
    GetSystemTimes(&sysIdle, &sysKernel, &sysUser);

    auto idle = subtractTimes(sysIdle, prevSysIdle);
    auto kernel = subtractTimes(sysKernel, prevSysKernel);
    auto user = subtractTimes(sysUser, prevSysUser);
    auto total = idle + kernel + user;
    
    prevSysIdle = sysIdle;
    prevSysKernel = sysKernel;
    prevSysUser = sysUser;
    
    return (kernel + user) * 100.0 / total;
#else
    // Linux 实现
    // 读取 /proc/stat
#endif
}

double getProcessCpuUsage() {
    // 获取当前进程 CPU 时间
```

## 10. 总结

故障排查核心要点：

| 问题类型 | 关键诊断工具 | 常见原因 |
|----------|--------------|----------|
| **连接问题** | WebSocket 日志、ICE 状态 | 服务器状态、防火墙、NAT 类型 |
| **音频问题** | WebRTC stats、音轨状态 | 权限、设备选择、编解码器 |
| **视频问题** | WebRTC stats、帧率监控 | 权限、网络带宽、编解码器 |
| **网络问题** | ping/traceroute、抓包 | 带宽不足、丢包、延迟高 |
| **资源问题** | 系统监控、内存检测 | 内存泄漏、CPU 瓶颈 |

**最佳实践**：

1. **先看日志**：90% 的问题可以通过日志定位
2. **检查基础**：网络、权限、设备状态
3. **使用工具**：chrome://webrtc-internals 是神器
4. **对比分析**：与正常状态对比找出差异
5. **记录问题**：建立问题知识库，避免重复排查
