### 1.2 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| **客户端框架** | C++17 + Qt 5.15+ | 跨平台支持、丰富的 UI 组件、良好的 WebRTC 集成 |
| **音视频引擎** | libwebrtc (原生) | 业界标准、功能完整、持续更新 |

**关键设计点**：

1. **Qt 信号槽机制**：UI 事件通过信号槽传递到业务层
   ```cpp
   // UI 事件到业务层
   connect(muteButton, &QPushButton::clicked, this, &MeetingPage::onMuteClicked);
   
   // 业务状态到 UI 更新
   connect(session, &Session::remoteVideoAdded, videoWidget, &VideoWidget::addRemoteVideo);
   ```

2. **视频渲染**：
   - 使用 `QVideoWidget` 或自定义 OpenGL 渲染
   - 支持动态布局（Grid/Focus 模式）
   - 帧率统计和延迟监控

3. **线程模型**：
   - UI 线程：仅负责界面渲染和用户交互
   - 工作线程：WebRTC 回调、媒体处理
   - 线程间通过 Qt 信号槽通信（自动队列化）

## 3. 关键设计决策

### 3.1 为什么选择 Mesh 拓扑？

**对比分析**：

| 特性 | Mesh | SFU | MCU |
|------|------|-----|-----|
| **架构复杂度** | 低 | 中 | 高 |
| **服务器成本** | 无需媒体服务器 | 需要中转服务器 | 需要混流服务器 |
| **客户端压力** | O(n-1) 上行，O(n-1) 下行 | O(1) 上行，O(n-1) 下行 | O(1) 上行，O(1) 下行 |
| **适用规模** | 2-6 人 | 10-50 人 | 50+ 人 |
| **延迟** | 最低 | 中等 | 较高 |

**选择理由**：

1. **产品定位**：技术面试展示、小规模团队会议（2-6 人）
2. **成本优势**：无需部署媒体服务器，降低运维成本
3. **学习价值**：完整展示 WebRTC P2P 连接过程
4. **性能可接受**：6 人会议总带宽需求 ≈ 15 Mbps（按 500kbps/路计算）

**Mesh 拓扑连接示意**：

```
        ┌─────────┐
        │ 用户 A  │
        └────┬────┘
         ╱   │   ╲
        ╱    │    ╲
   ┌───▼──┐ ┌▼───┐ ┌───▼──┐
   │用户 B│ │用户C│ │用户 D│
   └──────┘ └────┘ └──────┘
        ╲    │    ╱
         ╲   │   ╱
        ┌───▼───┐
        │ 用户 E│
        └───────┘

每个用户与其他 n-1 个用户建立 P2P 连接
```

### 3.2 为什么使用内存存储？

**设计考量**：

| 因素 | 内存存储 | 持久化存储 |
|------|----------|------------|
| **部署复杂度** | 无需数据库 | 需要数据库服务 |
| **响应速度** | 极快 | 较快 |
| **数据持久性** | 重启丢失 | 永久保存 |
| **扩展性** | 单实例 | 支持分布式 |

**选择理由**：

1. **无状态会议**：会议数据无需持久化
2. **简化部署**：一个 Node.js 进程即可运行
3. **面试展示**：突出核心 WebRTC 功能
4. **快速原型**：便于功能迭代

**存储结构**：

```javascript
// 内存数据结构
const rooms = new Map();      // roomId -> Room
const users = new Map();      // userId -> User
const connections = new Map(); // connectionId -> Connection

class Room {
    constructor(id, name, creator) {
        this.id = id;
        this.name = name;
        this.creator = creator;
        this.participants = new Map();  // userId -> Participant
        this.createdAt = Date.now();
        this.maxParticipants = 6;

```

### 3.3 为什么选择 Node.js 作为信令服务器？

**对比分析**：

| 平台 | 优势 | 劣势 |
|------|------|------|
| **Node.js** | 高并发、事件驱动、生态丰富 | 单线程 CPU 密集受限 |
| **Go** | 高性能、协程模型 | 学习成本较高 |
| **Python** | 简单易用 | 并发性能较弱 |
| **Java** | 企业级、成熟稳定 | 内存占用大 |

**选择理由**：

1. **WebSocket 原生支持**：ws 库性能优秀
2. **开发效率高**：代码简洁、迭代快速
3. **面试友好**：主流技术栈、易于理解
4. **部署简单**：单进程即可，可 PM2 扩展

**线程安全设计**：

```cpp
// WebRTC 回调 -> Qt 信号（跨线程安全）
class PeerConnectionObserver : public webrtc::PeerConnectionObserver {
public:
    void OnIceCandidate(const webrtc::IceCandidateInterface* candidate) override {
        // WebRTC 工作线程
        std::string sdp;
        candidate->ToString(&sdp);
        
        // 通过 Qt 信号槽传递到 UI 线程（自动队列化）
        emit iceCandidateReady(QString::fromStdString(sdp));
    }
    
signals:
    void iceCandidateReady(const QString& sdp);
};
```

GlobalConfig loadConfig() {
    GlobalConfig config;
    
    // 1. 默认值（已初始化）
    
    // 2. 配置文件
    if (QFile::exists("config.json")) {
        loadFromFile("config.json", config);
    }
    
    // 3. 环境变量
    if (qEnvironmentVariableIsSet("MINIMEETING_SIGNAL_SERVER")) {
        config.signalingServer = qgetenv("MINIMEETING_SIGNAL_SERVER");
    }
    
    // 4. 命令行参数
    QCommandLineParser parser;
    parser.addOption(QCommandLineOption("server", "Signaling server URL", "url"));
    if (parser.isSet("server")) {
        config.signalingServer = parser.value("server");
    }
    
    return config;
}

## 9. 总结

MiniMeeting 采用清晰的分层架构设计：

| 层级 | 核心职责 | 关键技术 |
|------|----------|----------|
| **UI 层** | 用户交互、界面渲染 | Qt Widgets、信号槽 |
| **会话层** | 会议管理、状态控制 | 状态机、业务逻辑 |
| **传输层** | 连接管理、信令处理 | WebRTC、WebSocket |
| **设备层** | 设备管理、媒体采集 | Qt Multimedia、系统 API |
| **媒体层** | 编解码、质量处理 | libwebrtc 内置模块 |

**设计亮点**：

1. **清晰的职责划分**：每层专注单一职责，便于测试和维护
2. **Qt 信号槽解耦**：层级间通过信号槽通信，降低耦合度
3. **线程安全设计**：UI 线程与工作线程明确分离
4. **Mesh 拓扑简化**：适合小规模会议，无需复杂媒体服务器
5. **可扩展架构**：预留插件接口，支持功能扩展
